      Mysql数据库优化

      Post on 2016-07-25

      主要内容

      • 数据库的演变过程
      • 数据库主从
      • 数据库双机热备
      • 数据库负载均衡
      • 数据库读写分离
      • 谈谈数据库的架构

      数据库的演变过程

      单主机模式

      最开始网站一般都是由典型的LAMP架构演变而来的，一般都是一台linux主机 包括apache服务器，php执行环境以及MySQL服务器，一般情况下，这些都在一台虚拟主机上，简称单主机模式

      缺点

      • web服务器和mysql服务器公用一台主机，共享硬件资源，可能存在某一方资源征用太大，导致整个应用产生瓶颈 {:&.zoomIn}
      • 当业务增长之后，没有办法做到横向扩展
      • 容错性太差，一旦主机存在问题，整个应用不可用

      独立主机模式

      随着业务的发展，可以把mysql服务器和web服务器主机分开，分别部署，就是独立主机模式。
      独立主机模式下，web服务器和mysql不再共享硬件资源，分别部署。没有把鸡蛋放在一个篮子里面，增加了容错性。如果只是mysql服务器故障，那么对于web上不访问服务器的应用是不会受到影响的。而且web服务器可以做到横向扩展，如果web服务器性能不够，可以增加多台web服务器，进行负载均衡，分散web服务器的压力

      缺点

      • 可扩展性问题：虽然web服务器可以做到横向扩展，但是mysql服务器是没有办法做到横向扩展的 {:&.bounceIn}
      • 可用性问题：mysql服务器存在单点问题，一旦mysql服务器宕机，对影响的影响很大
      • 性能问题：单台mysql服务器能够支撑的服务是有限的

      主从数据库 and 读写分离

      • 数据库的压力会越来越大，单数据库慢慢的就不能满足需求了，可以把mysql服务器分成主从数据库分担压力 {:&.zoomIn}
      • 一般网站对数据实时性要求不高，就会慢慢发展读写分离模式，对于普通的查询请求，分配到读库，对于修改请求，在主库上完成。对于读库，由于是无状态的，可以做到横向扩展。对于写库，还只能是单台主机

      好处

      • 实现服务器负载均衡 {:&.bounceIn}
      • 通过复制实现数据的异地备份
      • 提高数据库系统的可用性

      主从数据库

      工作原理是什么 ？？？

      • 主服务器凡运行语句,都产生一个二进制日志 binlog {:&.zoomIn}
      • 从服务器不断读取主服务器的binlog
      • 从主服务读取到的binlog,转换为自身可执行的relaylog
      • 执行relaylog

      实现步骤:

      • 首先确保主服务器打开二进制日志功能.这样,主服务器一旦有数据变化,立即产生二进制日志. {:&.zoomIn}
      • 从服务器也需要开启二进制日志和relay日志功能.这样可以从主服务器读取binlog,并产生relaylog
      • 在主服务器建立一个从服务器的账号,并授予权限
      • 指定从服务对应的主服务器,开启从服务器

      配置master主机:

      vim /etc/my.cnf
#开启二进制日志
log-bin=master-bin
#指定日志格式mixed/row/statement 磁盘row/语句
binlog-format=mixed
server-id=30

      配置slave主机:

      vim /etc/my.cnf
#中继日志
relay-log=slave-bin
server-id=31

      分别启动两台服务器

      service mysqld restart

      让两台服务器建立联系在 ，主服务器上创建相应的复制账号

      GRANT REPLICATION SLAVE,

REPLICATION CLIENT ON *.*

TO 'cuizhe'@'192.168.20.%'

IDENTIFIED BY 'cuijiji';

      在从服务器通过语句指定要复制的主服务器(注意,可以一主多从,不可一从多主).

      change master to master_host='192.168.20.30',
master_user='cuizhe',master_password='cuijiji',
master_log_file='master-bin.000001',
master_log_pos=285;
start slave;

      双机热备概念与作用

      • 通俗说法主主复制
      • 保持两个数据库的状态自动同步。对任何一个数据库的操作都自动应用到另外一个数据库，始终保持两个数据库数据一致
      • 可以做灾备，其中一个坏了可以切换到另一个
      • 可以做负载均衡，可以将请求分摊到其中任何一台上，提高网站吞吐量。 对于异地热备，尤其适合灾备

      如何解决主键冲突

      • 让1台服务器 1,3,5,7来增长
      • 另1台服务器 2,4,6,8来增长

      实现步骤

      分别从本次与全局进行修改

      一台服务器：

      set global auto_increment_increment = 2; 
set global auto_increment_offset = 1; 
set session auto_increment_increment = 2;
set session auto_increment_offset = 1;

      另一台服务器：

      set global auto_increment_increment = 2;
set global auto_increment_offset = 2; 
set session auto_increment_increment=2;
set session auto_increment_offset = 2;

      注：auto-increment-increment 和　auto-increment-offset　要写到配置文件　中，防止下次重启后失效

      问题又来了

      • 如果因为业务需要又增加一台服务器呢？
      • 如果因为业务需要又增加十台服务器呢？

      如果一个主数据库挂掉！ 咋整？

      • 只需要将一台服务器配置文件加入 read-only=1即可 {:&.zoomIn}
      • 挂了 注释掉即可使用！

      数据库的读写分离与负载均衡

      如何实现？

      • 一般用两种方法 {:&.zoomIn}
      • 1: 直接在PHP的mysql类做判断,最简单,不用额外加软件。比如discuz论坛.
      • 2: 用集群中间件.官方的mysql_proxy 比较流行的 kingshard
      • kingshard地址 https://github.com/flike/kingshard/blob/master/README_ZH.md

      首先利用mysql_proxy实现负载均衡

      • 下载安装mysql_proxy {:&.zoomIn}
      • wget http://ftp.ntu.edu.tw/pub/MySQL/Downloads/MySQL-Proxy/mysql-proxy-0.8.3-linux-glibc2.3-x86-64bit.tar.gz
      • 查看帮助 ./bin/mysql-proxy –help-all

      注意两点

      • proxy在哪个端口运行
      • 代理的是那几台服务器

      指定代理代理的服务器

      ./bin/mysql-proxy \
--proxy-backend-addresses=192.168.31.165:3306 \
--proxy-backend-addresses=192.168.31.145:3306

      测试说明一个问题

      负载是连接级别的 不是语句级别的！！！

      读写分离操作

      ./bin/mysql-proxy  \
--proxy-backend-addresses=192.168.31.165:3306 \ #配置写服务器
--proxy-read-only-backend-addresses=192.168.31.145:3306 \ #配置读服务器
--proxy-lua-script=mysql-proxy/share/doc/mysql-proxy/rw-splitting.lua #利用这个脚本

      简写

      ./bin/mysql-proxy -b 192.168.31.165:3306 \
-r 192.168.31.145:3306 \
-s /root/mysql-proxy-0.8.3-linux-glibc2.3-x86-64bit/share/doc/mysql-proxy/rw-splitting.lua
